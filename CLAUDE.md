# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Bank AI — a polished, phone-width React chat app for a fictional bank, all running off a mock API (no real backend or LLM). Three routes:

- `/` (`WelcomePage`) — a spinning orb, brand mark, "What can I help?", and a grid of topic cards loaded from a mock API. Submitting the chat input starts a fresh conversation and navigates to `/chat`.
- `/chat` (`ChatPage`) — the conversation view. The thread is chosen from the URL: `?topic=<id>` resumes/starts a topic thread (with a bespoke welcome screen when empty), otherwise it's the `general` thread. A `?message=` deeplink seeds the general thread once.
- `/docs` (`DocsPage`) — a URL-only living-documentation + playground page for the reply-authoring system (no nav link points to it). See the **Docs / playground** bullet under Architecture.

Assistant replies are **canned markdown strings** returned by MSW; the frontend fakes streaming by typewriter-revealing them. Replies can embed rich UI — GFM tables/task-lists, inline colored highlights, and code-split "custom blocks" (charts, action cards, follow-up suggestion pills) — driven entirely by the reply text.

## Monorepo layout

This is an **npm-workspaces monorepo** (`"workspaces": ["apps/*", "packages/*"]` in the root `package.json`; npm 7+, no pnpm/Yarn/Turborepo). Five packages:

- **`apps/web`** (`@bank-ai/web`) — the React + Vite frontend. Everything the doc below calls `src/…`, `public/…`, `vite.config.ts`, `tsconfig.app.json`, `tsconfig.node.json`, `index.html` now lives under `apps/web/`.
- **`apps/server`** (`@bank-ai/server`) — the Hono backend (real-LLM mode). What the doc calls `server/…` now lives directly under `apps/server/` (the files are flat there, not under a nested `server/`). Its `.env`/`.env.example` (OpenRouter key, `PORT`) live here.
- **`apps/blocks-spend`** (`@bank-ai/blocks-spend`) — a **Module Federation remote** (Vite, port 9998) that owns the spend blocks (`spendTrend`/`spendDonut`/`spendBreakdown`). It exposes each block as `{ schema, Component }`, plus a `./docs` module, and ships its own self-contained Tailwind CSS. The web host loads these blocks at runtime from a backend manifest — so this package deploys on its own cadence. See the **Federated block remotes** bullet under Architecture.
- **`packages/shared`** (`@bank-ai/shared`) — the **contract layer** both apps import: `src/types.ts` (domain types like `ThreadId`, `Topic`, `Insight`, `BlockRemoteManifest`) and **every block's Zod schema** at `src/blocks/<name>.ts`, re-exported from a barrel `src/index.ts`. It's consumed as **raw TS source** (no build step): Vite resolves it for web, Node's native type-stripping resolves it for the server. Its `exports` map points at `./src/index.ts`, and the barrel's relative imports carry explicit `.ts` extensions so Node's runtime ESM resolver can load them.
- **`packages/blocks-kit`** (`@bank-ai/blocks-kit`) — the shared **presentational toolkit** for blocks (`BlockCard`, `BlockFallback`, `ComponentSkeleton`, `ACCENTS` from `colors`, and the `BlockDoc` type). Consumed as **raw TS/TSX source** (no build step, like `@bank-ai/shared`) by both the host's local blocks and the block remotes, so a block looks identical wherever it's built. React is a peer dep.

Scripts run from the repo root and delegate to workspaces (`npm run dev`, `dev:mfe`, `dev:real`, `server`, `remote`, `build`, `lint`, `preview`). `npm run build` runs `tsc -b` (project references cover all five packages) then `vite build` (the web host; the remote builds via `npm run build -w @bank-ai/blocks-spend`). When adding a new frontend dep, add it to `apps/web/package.json`; server deps to `apps/server/package.json`; spend-remote deps to `apps/blocks-spend/package.json`. `.env.real` (`VITE_API_MODE=real`, read by Vite) stays in `apps/web`.

## Commands

```bash
npm run dev      # Vite dev server (mock API) on http://localhost:9999 (port pinned in apps/web/vite.config.ts)
npm run dev:mfe  # spend remote (:9998) + Vite (:9999, mock API) — mock backend with federated blocks
npm run dev:real # Hono backend (:8787) + spend remote (:9998) + Vite (:9999, MSW off), via concurrently
npm run server   # just the Hono backend (@bank-ai/server)
npm run remote   # just the spend block remote (@bank-ai/blocks-spend, :9998)
npm run build    # tsc -b (typecheck all packages) then vite build → dist/
npm run lint     # oxlint (NOT eslint)
npm run preview  # serve the production build
```

No test runner is configured. `npm run build` is the typecheck gate — run it to validate changes. The spend blocks live in the remote, so `npm run dev` alone (without the remote on :9998) renders them as the inline fallback — use `dev:mfe` or `dev:real` to exercise them.

## Architecture

- **Stack**: Vite 8 + React 19 + TypeScript, Tailwind CSS **v4**, React Router 7, TanStack Query 5 (server data), Zustand 5 (chat state), MSW 2 (mock API), Zod 4 (block validation), Recharts 3 (charts), react-markdown + remark-gfm + remark-directive (rich replies), sonner (toasts), lucide-react icons.

- **Tailwind v4 is CSS-first** — there is no `tailwind.config.js`. The plugin is wired in `vite.config.ts` (`@tailwindcss/vite`). All theme tokens, keyframes, and custom utilities live in `src/index.css` inside `@theme`, `@property`, `@keyframes`, and `@utility` blocks. Brand colors are `--color-accent-1/2/3`; inline-highlight tones are `--color-tone-positive/negative/warning/info`; surfaces are `--color-ink` / `--color-ink-deep`. Use them via classes like `from-accent-1` / `text-accent-3` / `bg-ink-deep`.

- **App bootstrap** (`src/main.tsx`): MSW is started **before** React renders, and only in `import.meta.env.DEV`, so production builds never load the mock worker. React mounts inside `QueryClientProvider` → `BrowserRouter`. `App.tsx` is the route table plus the always-on `AnimatedBackground` (drifting ambient orbs) and the global sonner `Toaster` (styled glassmorphism, offset to clear the bottom bar).

- **Mock API (MSW)**: The app makes real `fetch`es that MSW intercepts at the Service Worker layer (`public/mockServiceWorker.js`, generated by `npx msw init`). Handlers and all seed/reply data live in `src/mocks/handlers.ts`; the worker is set up in `src/mocks/browser.ts`. Endpoints:
  - `GET /api/topics` → topic cards (`delay(600)` so loading skeletons show)
  - `GET /api/insights` → insight cards for the Insights welcome screen
  - `GET /api/block-remotes` → the `BlockRemoteManifest` (`{ remotes: [{ name, entry, blocks[] }] }`) telling the host which Module Federation remotes to register and where (`delay(150)`). Mirrors the Hono server's `/api/block-remotes`; both point at the local spend remote (`http://localhost:9998/remoteEntry.js`).
  - `POST /api/chat` `{ threadId, message }` → `{ reply }`, a full canned markdown string keyed by thread (`delay(700)`). On the `insights` thread the message text is matched against per-insight replies first.

- **Chat state (`src/store/chatStore.ts`, Zustand + persist)**: One `Message[]` per thread, keyed by `ThreadId` (`transfer | cards | savings | security | insights | general`), plus a single `pending` thread id that locks that thread's input while a reply is in flight or streaming. `sendMessage` appends the user message and kicks off `requestReply`; `startGeneralFresh` resets the general thread from one message (used by the home input and `?message=` deeplink); `retryLast` re-requests after an error; `finishStreaming` is called by the typewriter when a reply finishes revealing (clears `pending`, drops `status`). Persisted to **sessionStorage** as `bank-ai-chat` — only `threads` are persisted, and any message left mid-stream is normalized to done so it renders fully on reload.

- **Fake streaming**: `sendChat` returns the whole reply at once. `MessageBubble` marks the new assistant message `status: 'streaming'` and feeds its prose through `useTypewriter` (`src/hooks/useTypewriter.ts`), which reveals ~2 chars/16ms. Atomic spans — `` ```bank:<name>``` `` fences and `:hl[…]{…}` directives — are snapped past in one frame so their raw syntax never flashes. On completion it calls `finishStreaming`.

- **Layout (`src/components/AppShell.tsx`)**: the phone-width column (`mx-auto`, centered, `h-dvh`) shared by both pages — optional header, a scrollable content area, and the `ChatInput` + `Footer` at the bottom. Two modes:
  - default (home): header/content/bottom-bar in normal flex flow.
  - `overlay` (chat): content fills the height and scrolls behind blurred, fading top/bottom bars; the bars' heights are measured with a `ResizeObserver` and fed into the scroll area's padding + `scroll-padding` so the first/last message clears the chrome and scroll-to-bottom lands above the input.

- **Topics → welcome screens**: `src/api/topics.ts` → `src/hooks/useTopics.ts` (TanStack Query) → `src/components/TopicGrid.tsx` (skeletons while `isPending`, inline retry on `isError`, `TopicCard`s on success). Selecting a card navigates to `/chat?topic=<id>`. When a topic thread is empty, `ChatPage` shows a bespoke **welcome screen** resolved in `src/components/welcome/index.tsx` (`TransferWelcome`, `CardServicesWelcome`, `SavingsWelcome`, `SecurityWelcome`, `InsightsWelcome`, falling back to `GenericWelcome`); each takes `{ topic, onSend }` and offers quick-action prompts. `InsightsWelcome` fetches `/api/insights` (`useInsights`) and renders insight cards whose tap sends the insight title as the first message.

- **Rich assistant replies (`src/components/Markdown.tsx`)**: renders reply markdown with `remark-gfm` (tables, task lists, strikethrough, autolinks). **Raw HTML is intentionally not rendered** (no `rehype-raw`), so reply content can't inject markup. Two custom authoring affordances ride on top:
  - **Inline highlights** — `:hl[text]{tone=positive|negative|warning|info}` (a remark text directive mapped to `<Highlight>`; unknown tone → plain text, missing → `info`).
  - **Custom blocks** — `` ```bank:<name>``` `` fenced code whose body is JSON. `CustomBlock` (`src/components/blocks/CustomBlock.tsx`) looks `<name>` up in the **block registry**, `JSON.parse`s + Zod-validates the body, and renders a lazily-imported component; any failure (unknown name, bad JSON, schema mismatch, render error) degrades to an inline `BlockFallback` notice.

- **Block registry (`src/components/blocks/`)**: each block's **Zod schema lives in `packages/shared/src/blocks/<name>.ts`** (shared with the server; see Monorepo layout) — imported via `@bank-ai/shared`. Everything else is a **self-contained folder** `blocks/<name>/` owning: `<Name>.tsx` (default-export component, imports its `<Name>Data` type from `@bank-ai/shared`), `index.ts` (the `defineBlock(schema, () => import('./<Name>'))` wiring — imports `<name>Schema` from `@bank-ai/shared` — default-exported), `docs.ts` (gallery/playground metadata — see the Docs bullet), and an optional `guide.md` (long-form prose). `defineBlock` lives in `defineBlock.ts` and ties the component's `data` prop to the schema's inferred type at compile time while code-splitting the component (keeping Recharts out of the initial bundle). `registry.ts` is a **glob aggregator** — `import.meta.glob('./*/index.ts', { eager: true })` keyed by folder name — so it's written once and never edited. The folder name = block name = `bank:<name>` fence name. Eager-globbing `index.ts` pulls each block's (small) schema into the chat bundle but leaves components lazy. `defineBlock` returns a `kind:'local'` variant of `BlockDefinition` — the registry value is a discriminated union of `kind:'local'` (compiled-in) and `kind:'remote'` (federated, see next bullet), and `CustomBlock` branches on it. Local blocks: `allocationDonut`, `driftBars` (Recharts), `actionCard`, `suggestions`, `wizard`. The spend blocks (`spendTrend`/`spendDonut`/`spendBreakdown`) are **federated** — they live in the `@bank-ai/blocks-spend` remote, not in `apps/web`. **Adding a local block = add the schema at `packages/shared/src/blocks/<name>.ts` + re-export it from the shared barrel, then create one `blocks/<name>/` folder (component + index + docs) under `apps/web`; no edits to the registry or any other central file.** Shared **presentational** infra — `BlockCard`, `BlockFallback`, `ComponentSkeleton`, `ACCENTS` (`colors`), and the `BlockDoc` type — lives in **`@bank-ai/blocks-kit`** (so remotes share it); host-only machinery (`BlockErrorBoundary`, `ChatThreadContext`, `CustomBlock`, `registry`, `defineBlock`) stays at the `blocks/` root. Hard rule: a block's `index.ts` must **never** import its sibling `docs.ts`/`guide.md`, so docs prose stays out of the chat bundle. The `suggestions` block is special-cased: `MessageBubble` uses `blocks/suggestions/splitSuggestions` to hoist it out of the prose and render it (follow-up pills) below the message actions, never inline; pills dispatch back into the current thread via `ChatThreadContext`. The `wizard` block publishes itself to a Zustand **block bus** (`src/store/blockBus.ts`) so a `signal` suggestion pill can reopen it; its drawer (`blocks/wizard/WizardDrawer`) is mounted via `BlockOverlayHost` on the chat page only. (`apps/server/blocks.ts` imports those block schemas from `@bank-ai/shared` for the model-facing catalog — keep the shared barrel's exports in sync when renaming a block.)

- **Federated block remotes (Module Federation)**: blocks can also be **owned by a separate package and loaded at runtime**, so a team ships blocks without a host release. Uses **`@module-federation/vite`** (Vite 8-compatible) with its **runtime API** (`registerRemotes`/`loadRemote` from `@module-federation/runtime`). Remotes are **not** listed statically in `apps/web/vite.config.ts` — the host `federation()` plugin just initializes the MF runtime + shared scope (React, `react-dom`, `recharts`, `zod` as **singletons**, so remotes reuse the host's copies rather than bundling their own). At boot, `main.tsx` fetches `GET /api/block-remotes` (the `BlockRemoteManifest`, served by both MSW and the Hono server) and calls `registerRemoteBlocks(manifest)` (`blocks/registry.ts`), which `registerRemotes(...)` each remote and adds a `kind:'remote'` `BlockDefinition` per advertised block whose `load()` `loadRemote`s the exposed `{ schema, Component }` (memoized). `CustomBlock` renders those via a `<RemoteBlock>` that suspends on the load, then Zod-validates against the **remote-provided** schema before rendering — so the host needs **no compile-time dependency** on a remote block's data shape, and any failure still degrades to `BlockFallback`. The reference remote is **`apps/blocks-spend`** (see Monorepo layout): its `vite.config.ts` `exposes` each block (`./spendTrend` → `./src/spend-trend.ts`, etc.) as `{ schema, Component }` plus `./docs`, ships self-contained Tailwind (`src/index.css` imports `tailwindcss/theme` + `tailwindcss/utilities` **without preflight** so the injected stylesheet doesn't clobber the host), and imports `ACCENTS`/`BlockCard` from `@bank-ai/blocks-kit`. **Two gotchas** (both in code comments): `registerRemotes` needs `type: 'module'` because the Vite plugin serves `remoteEntry.js` as ESM; and both `federation()` configs set `dts: false` (cross-boundary types aren't generated — validation is runtime Zod). **Adding a new remote = add it to the manifest (both backends) and register it; adding a block to an existing remote = expose it there — zero host changes.** (Not yet wired: a prod/CDN manifest — the entry URL is env-configurable via `BLOCKS_SPEND_ENTRY` on the Hono server but currently points at the local `:9998` remote.)

- **Docs / playground (`/docs`, `src/pages/DocsPage.tsx` + `src/components/docs/`)**: a URL-only living-documentation page with two tabs (tracked in `?tab=gallery|playground`), lazy-loaded so its content stays out of the main bundle. The **Gallery** iterates `Object.keys(blockRegistry)` (the registry is the source of truth, so a newly registered block — local or federated — can't be silently omitted; a block with no docs at all renders a "documentation pending" placeholder), with a **search box + category facets** over the docs metadata and **on-demand previews**: each block is a collapsed card whose live preview, examples, source, and `guide.md` render only on expand, so opening the gallery never mounts every block (or its lazy chunk) at once — it scales to many blocks. The **Playground** is a side-by-side markdown editor + live preview (reusing `Markdown` directly, no typewriter) with a **searchable** insert palette (inserts a block's first example), starter docs, copy, and a base64-encoded `?md=` shareable URL. Both tabs render blocks under `<ChatThreadProvider value={null}>` so the page is **side-effect free**: charts/cards render fully, while `prompt`/`signal` pills and the wizard drawer (which would post into a real chat thread) stay inert previews. **Per-local-block doc metadata lives in each block's `blocks/<name>/docs.ts`** (a `BlockDoc`: `{ title, summary, category, keywords?, examples[] }`, where `examples[].data` is the raw block object; the `BlockDoc` type lives in `@bank-ai/blocks-kit`, re-exported from `blockDocs.ts`), aggregated by `src/components/docs/blockDocs.ts` via `import.meta.glob` (which also globs the optional `guide.md` files as raw markdown into `blockGuides`). **Federated blocks supply their docs via the remote's `./docs` expose** (a `Record<name, BlockDoc>`); `useBlockDocs` (`src/components/docs/useBlockDocs.ts`) `loadRemote`s each remote's `./docs` at runtime and merges it with the local glob, and **both** the Gallery and Playground read from it (so remote blocks show full docs + insert-palette pills). Plain-markdown samples, insert snippets, and starter docs live in `samples.ts` (cross-cutting, not per-block). Keep examples MYR/Malaysia-flavored. The dependency direction is one-way (docs → blocks): the aggregator (and `samples.ts`) is imported only by the lazy `/docs` route, and per-block `index.ts` never imports `docs.ts`/`guide.md`, so docs prose never lands in the main bundle.

- **`Orb`** is reused at two sizes via a `size` prop (large on the home page, compact inside the chat input). Its spin and the chat input's rotating gradient border are both driven by the registered `@property --angle` custom property animated with the `spin-angle` keyframe; `ChatInput` speeds up the rotation and activates the orb on focus.

- **`Message.icon`-style string-keyed maps**: several types carry a string `icon`/`tone` key resolved to a component/class at render time — `Topic['icon']` → `ICONS` in `TopicCard.tsx`, `Insight['icon']`/`tone` → `InsightCard.tsx`. Adding an option means extending both the union in `packages/shared/src/types.ts` and the corresponding map.

## Locale & currency

Bank AI is a **Malaysian** bank — design everything from a Malaysia / MYR perspective unless a scenario is explicitly international.

- **Currency**: all monetary amounts are in Ringgit Malaysia, formatted as `RMx,xxx` (the `RM` prefix immediately followed by the number, no space) — e.g. `RM2,140`, `RM89.99`, `RM15`. Never use `£` or `$`.
- **Products & terminology**: prefer Malaysian equivalents — EPF (incl. voluntary i-Saraan), PRS (Private Retirement Scheme, tax relief), ASB / ASNB funds, unit trusts, fixed deposits, easy-access savings. Avoid UK/US-specific products (ISA, Lifetime ISA, 401k, etc.).
- **Transfers**: DuitNow (instant) and IBG / Interbank GIRO for local rails; SWIFT only for international transfers.
- **Conventions**: `DD/MM` dates, one-time passcodes referred to as TAC, and Malaysian names/merchants in mock data.
- **International scenarios**: only depart from MYR/Malaysia when the case is genuinely cross-border (e.g. an outbound SWIFT transfer), and make that explicit.

## Conventions

- Components are named exports in `src/components/` (blocks under `components/blocks/`, welcome screens under `components/welcome/`); pages in `src/pages/`. Block components are **default** exports (required by the lazy-import loader). Path-less relative imports throughout (no path aliases configured).
- Keep styling in Tailwind utility classes; reach for `src/index.css` only for theme tokens, keyframes, and `@utility` definitions that can't be expressed inline.
- The look is dark glassmorphism: translucent `white/5–10` surfaces, `border-white/10`, `backdrop-blur`, accent gradients. Match it when adding UI.
